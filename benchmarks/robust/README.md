# Robust Scripting Benchmark

A redesign of the LLM-scripting benchmark that fixes the flaws that made the
original produce ~9.5±0.4 noise where any ranking was an artifact of setup
choices rather than a real difference.

## What was wrong before, and what changed

| Problem in the old benchmark | Fix here |
| --- | --- |
| Comparators were strawmen (idiomatic prose dumped into boilerplate) | **Hand-authored, idiomatic artifacts** for every system, all written with equal care, committed under `artifacts/` for review, not generated by the harness at runtime. |
| Scenarios only exercised the happy path, so every system produced the same outcome | **16 branch-forcing scenarios** that each trigger specific conditional paths (missing input, `main` confirmation, low coverage, deploy failure, health exhaustion, regenerate, kebab-case rejection, each deploy target, custom runtime). |
| Score = LLM judge's holistic vibe on the executor's self-report | **Deterministic checklist (primary):** each scenario lists observable behaviors; a code grader checks the produced event trace. The judge is **secondary**. |
| Executor self-report style leaked into the score ("logic exists" → credit) | Executor emits a **typed event log** and is told to emit events **only for actions actually performed in this run**. Untriggered branches earn nothing; the checklist credits behavior, not claims. |
| Executor and judge were the same model family (self-grading) | **Separate models:** `claude-haiku` executes, `claude-sonnet` judges. |
| MDScript was judged without its execution spec, unlike self-describing code | **MDScript executors read `README.md`** (honoring the `read` directive); the directive exists precisely to teach an agent how to run MDScript. Code artifacts need no external spec. |
| Judge could be swayed by framework reputation | **Blind judging:** the judge sees only the produced trace, never the artifact, the language, or the system name. |

## Layout

```
artifacts/<case>/<system>.<ext>   # the reviewable workflows under test (12 files)
scenarios/<case>.json             # branch-forcing scenarios + required/forbidden behaviors
build_harness.py                  # generates harness.js from the scenario JSON (no drift)
harness.js                        # the workflow: execute → grade (code) + judge (blind)
results/latest.json               # full per-run data (traces, checklist detail, judge notes)
results/latest-summary.md         # compact tables
```

Cases: `deploy_branch`, `release_notes`, `onboard_service`.
Systems: `mdscript`, `guidance` (idiomatic), `lmql` (idiomatic), `ell` (idiomatic).

## How a run works

1. **Execute.** For each (case, system, scenario, repeat) a haiku agent reads the
   artifact file (MDScript also reads the spec), follows the scenario's fixed
   `world` (mocked command results and user answers, consumed in order), and
   emits a `TRACE`: a typed event log (`ASK_USER`, `REQUEST_CONFIRMATION`,
   `RUN_COMMAND`, `WRITE_FILE`, `NOTIFY`, `ROLLBACK`, `HEALTH_CHECK`,
   `ENTER_STATE`, `GIT_INIT`, `ABORT`, `COMPLETE`) plus an `outputs` object.

2. **Score (primary, deterministic).** A pure-JS grader evaluates each scenario's
   `checks` against the trace: `event` / `event_absent` / `count` /
   `output` / `output_absent`. Coverage = passed / total. No model involved.

3. **Score (secondary, blind).** A sonnet judge scores the trace 1-10 on whether
   the run accomplished the scenario, seeing only the trace.

4. **Aggregate.** Per-system checklist % and judge mean, per-case breakdown, and
   **disagreements** (checklist passed but judge harsh, or vice versa), the
   places worth reading by hand.

## Honest limitations

- **Execution is still stochastic; only the grading is deterministic.** haiku
  *simulates* running each artifact. `REPEATS` and the checklist sd column exist
  to expose that variance. Some sub-100% results are a single flaky repeat, not
  an artifact defect. n=2 per scenario is small; raise `REPEATS` for tighter CIs.
- **16 scenarios is a starting set,** not exhaustive coverage of every branch.
- The checklist measures *behavior performed*, not code quality or runtime
  correctness of the artifacts themselves. That's what the secondary judge and
  the disagreement list are for.

## Reproduce

```bash
python3 benchmarks/robust/build_harness.py      # regenerate harness.js from scenarios/
# then run harness.js via the workflow runner (executor=haiku, judge=sonnet)
```

Edit the `CASES_TO_RUN` / `REPEATS` knobs at the top of `harness.js` to scale.
See `results/latest-summary.md` for the latest numbers.
